01 / Strengths

What Spotify gets right.

🎯
Daily Mix & Discover Weekly
Best-in-class personalization. Low effort, high perceived value, delivered on a predictable schedule users build habits around. The fact that millions of people plan their Monday morning around Discover Weekly refreshing is a product achievement worth studying. This is the benchmark for music app personalization.
✓ Benchmark feature
🗺️
Made For You hub
Discovery has a permanent, predictable home instead of being buried or reshuffled. Users know exactly where to go. This sounds obvious — most apps still don't do it. The hub creates a consistent mental model that actually makes personalization feel intentional rather than random.
✓ Strong wayfinding
📱
Cross-device session handoff
Close the laptop, open your phone — it's still playing at the exact position. This is a genuine engineering and product achievement that users have never consciously noticed, which is exactly the mark of excellent UX. When a feature is invisible it's working perfectly.
✓ Invisible excellence
02 / The problem

Where Spotify creates friction.

Spotify's biggest gap isn't personalization — it's intent. The app is excellent at surfacing what you might like, but poor at understanding what you're trying to do right now. Those are different problems and they require different solutions.

01
Search intent is ambiguous
Search "true crime" and you get podcasts, playlists, and audiobooks mixed together. Spotify guesses your intent and guesses wrong often enough that users develop workarounds — adding words like "podcast" or "playlist" to force the right result type. That's the user doing work the interface should be doing.
02
The home screen is unstable
The layout reshuffles constantly. Users lose the ability to build a mental map of where things live. A home screen's job is to be predictably navigable. When it moves too often it stops being navigation and becomes discovery — whether the user wanted discovery or not.
03
Algorithms lack explainability
When Discover Weekly surfaces a track there's no signal about why. Users who love the pick have no way to reinforce that signal beyond a save. Users who want to understand the pattern can't. This limits long-term trust and makes the algorithm feel like a black box rather than a collaborator.
Deep dive — Problem 01
The search intent problem in detail

When a user types "true crime" they have a specific intent — podcast, playlist, or audiobook — but Spotify surfaces all three content types in one mixed feed. The user then has to scroll and filter, doing work the product should have handled upfront.

The workaround users have developed is telling: they type "true crime podcast" or "true crime playlist" to force a result type. This is a clear signal that the interface is making users compensate for a missing feature. Every time a user adds a qualifier word to a search, that's a product failure hiding in plain sight.

04 / The fix

Three things I'd build.

Each fix targets a specific user behavior that indicates friction — the workaround, the complaint, the churn signal. The common thread: reduce the work the user has to do to get the experience they want.

🔧
Persistent content-type filter
Set once, applied everywhere. Music / Podcasts / Audiobooks as a user preference that persists across sessions. Not a per-search toggle — a saved setting that follows you. Reduces the cognitive load on every subsequent search session. This is the highest-leverage fix on the list.
💬
"Why this track" explainability
A lightweight label on algorithmic picks: "Because you listen to The National" or "Fans of Bon Iver also saved this." Similar to what TikTok does with its For You signals. Low build cost, meaningfully higher trust. Users who understand why a recommendation appeared are more likely to engage with and train the algorithm.
📌
Pinned home layout row
An opt-in stable section at the top of the home screen for power users who want predictability over surprise. The algorithm still runs everywhere else — this just gives one row a fixed position. Targets a real segment that currently adds qualifiers to searches or churns to Apple Music for the library-first experience.
The test I apply to every product decision: is the user compensating for something the interface should handle? If yes, that's a product problem worth fixing. All three fixes above start from a real user workaround.
05 / Prioritization

What I'd ship first and why.

Prioritizing by whether the fix addresses an active user workaround. The filter and explainability tag both do. The layout option solves a real problem for a smaller segment — worth validating demand before committing build time.

Idea User impact Build cost Addresses workaround? My call
Persistent search intent filter High Low Yes — users type qualifier words Ship first
"Why this track" explainability High Medium Yes — users can't train the algorithm Ship second
Pinned home layout row Medium Medium Partially — some users churn to Apple Music Test demand first
📡
How I prioritize
I ask one question first: is the user already doing something to work around this? If yes, that's validated demand — the problem is real and users care enough to compensate for it. The search filter has the clearest workaround signal (users add qualifier words to searches), which is why it goes first. The layout option has a real but smaller signal (churn to Apple Music), which is why I'd test demand before building.
