Room 03 / Decision Journal

Small decisions that reveal
how I think.

Should the BRQ wizard autosave progress between sessions?
BRQ Project ▾
Clients sometimes start the intake and get pulled away before finishing. We were losing partially-completed forms with no recovery path.
Options considered
Add browser localStorage autosave. Build server-side session persistence. Ship a shorter flow that can be completed in one sitting. Do nothing.
Decision
Defer autosave. Design the flow to complete in under 8 minutes and remove optional fields that cause hesitation. Ship first, add draft recovery if analytics show meaningful abandonment.
What I'd revisit
If usage data showed 20%+ abandonment mid-flow, autosave becomes the priority fix. Without that data, it's premature complexity.
Principle: Complexity earns its place. Ship the simpler version first and let data justify the harder build.
Should clients see technical ERP and integration fields upfront?
BRQ Project ▾
The original BRQ asked technical AR code and ERP questions early, before the client had context for why we needed them. IT stakeholders answered incorrectly; non-technical stakeholders skipped entirely.
Options considered
Show all fields to all clients. Hide technical fields entirely and let PMs fill them separately. Reveal technical fields conditionally based on institution type.
Decision
Conditional reveal. Community colleges and small private institutions see a simplified flow. Multi-campus systems and large universities see the full technical section. Institution type drives the branching logic.
What I'd revisit
The branching adds maintenance complexity. Worth monitoring whether the split actually reduces follow-up questions or just shifts where they happen.
Principle: The client should not feel punished for not knowing implementation language. That's our job to translate.
Should Spotify add a pinned home layout option for power users?
Spotify ▾
Power users frequently complain that the Spotify home screen rearranges itself, making it harder to navigate to the same content reliably. The algorithmic home is optimized for discovery, not navigation.
Options considered
Full user-controlled home layout. A pinned "favorites" row at the top that stays stable. A toggle between "discovery mode" and "navigation mode." Do nothing.
Decision
Test demand first with a pinned row — not a full layout editor. A single pinned section adds predictability for power users without committing to preference management infrastructure for a feature with uncertain adoption.
What I'd revisit
If adoption is high and users are actively managing the pinned section, invest in a fuller layout customization system. If adoption is low, deprioritize and focus on search intent fixes instead.
Principle: Validate demand with the smallest version of the feature before building the infrastructure to support it.
Should AI-powered recommendations show reasoning to users?
Product Thinking ▾
Across both Spotify (why is this in my Discover Weekly?) and the BRQ tool (why is this field appearing now?), users encounter AI or algorithmic decisions without explanation. The question is whether transparency helps or hurts the product experience.
Options considered
No explanation — keep the interface clean. Full explanation — show the reasoning algorithm. Light explainability — a brief tag or label. User-controlled — let users toggle explanation on/off.
Decision
Light explainability by default. A brief "because you listened to X" or "this field appears for multi-campus systems" gives enough context to build trust without overwhelming the interface. Users who want more can ask.
What I'd revisit
If users are actively ignoring the explainability labels, reconsider whether the cognitive cost is worth the trust benefit. Run a test.
Principle: Trust depends on evidence. A recommendation without explanation increases speed but decreases confidence. The tradeoff is worth making explicit.
When a PM inherits a stalled implementation, what should they prioritize first?
PM Judgment ▾
I inherited a stalled higher education implementation mid-project with unresolved ERP issues, a frustrated client, and unclear internal ownership. The immediate instinct was to fix the technical problem. That was the wrong instinct.
Options considered
Fix the technical ERP issue immediately. Clarify internal ownership first. Rebuild the client relationship first. Do a full audit before doing anything.
Decision
Audit first, client relationship second, technical fix third. The ERP issue was blocking delivery, but a frustrated client won't trust any fix until they believe someone is actually in charge. Establishing that trust bought us the time to solve the technical problem correctly.
What I'd revisit
If the technical issue had been causing active client data problems, I'd have reordered — fix the harm first, then rebuild the relationship. Context matters more than any rule.
Principle: When a project is stalled, the blocker is almost never just technical. Fix the trust first.
How I approach decisions
Every decision in this journal follows the same structure: what options did I actually consider (not just the one I chose), what did I decide and why, and what would change my mind. The third column is the most important one. A decision without a "what would make me revisit this" is just a guess with extra confidence.
